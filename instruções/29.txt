# Event Architecture
Version: 1.0

---

# Objetivo

Definir a arquitetura orientada a eventos do Conecta Flow.

Este documento estabelece:

- tipos de eventos;
- estrutura;
- responsabilidades;
- produtores;
- consumidores;
- regras de processamento.

---

# Filosofia

Eventos representam fatos que aconteceram.

Um evento não representa uma intenção.

Correto:

"conversation.message_received"

Errado:

"process_customer_message"

---

# Princípio Fundamental

O passado não pode ser alterado.

Eventos representam histórico.

Estados atuais representam consequência.

---

# Exemplo

Evento:

message_received


Gera:

- atualização da conversa;
- análise da IA;
- classificação;
- notificações;
- métricas.

---

# Tipos de Eventos

Existem três categorias.

---

# 1. Domain Events

Eventos internos do negócio.

Representam mudanças importantes.

Exemplos:

conversation.created

lead.qualified

handoff.requested

appointment.created

---

# 2. Integration Events

Eventos vindos ou enviados para sistemas externos.

Exemplos:

whatsapp.message_received

instagram.message_received

payment.confirmed

---

# 3. System Events

Eventos operacionais.

Exemplos:

integration.failed

workflow.completed

backup.finished

---

# Estrutura Padrão

Todo evento deve possuir:

```

{
 event_id,
 event_type,
 version,
 timestamp,
 company_id,
 actor,
 source,
 correlation_id,
 payload
}

```

---

# event_id

Identificador único.

Utilizado para:

- auditoria;
- idempotência;
- rastreamento.

---

# event_type

Nome do evento.

Formato:

dominio.acao

Exemplos:

conversation.created

lead.scored

agent.handoff_requested

---

# version

Versão do contrato.

Exemplo:

1.0

Permite evolução sem quebrar consumidores.

---

# timestamp

Momento da criação.

Formato UTC.

---

# company_id

Obrigatório quando relacionado a uma empresa.

Garante isolamento.

---

# actor

Quem originou.

Exemplos:

user

agent

system

integration

---

# source

Origem.

Exemplos:

webapp

n8n

whatsapp

instagram

scheduler

---

# correlation_id

Permite acompanhar uma cadeia completa.

Exemplo:

Mensagem recebida.

↓

IA

↓

Handoff

↓

Notificação

---

# Processamento

Eventos devem ser:

- pequenos;
- claros;
- imutáveis;
- versionados.

---

# Idempotência

Todo consumidor deve assumir que um evento pode chegar mais de uma vez.

---

# Exemplo

Mensagem duplicada recebida.

Sistema deve verificar:

event_id

provider_message_id

antes de criar registro.

---

# Persistência

Eventos importantes devem ser armazenados.

Modelo:

event_store

---

Campos:

id

event_type

company_id

payload

created_at

processed_at

---

# Retenção

Eventos críticos:

retenção longa.

Eventos operacionais:

retenção configurável.

---

# Realtime

Eventos podem alimentar:

Supabase Realtime.

Exemplo:

nova mensagem.

---

# n8n

O n8n consome eventos quando:

existe automação externa;

processo assíncrono;

integração.

---

# n8n nunca deve ser fonte de verdade.

O evento nasce no domínio.

---

# IA

A IA reage a eventos.

Nunca monitora banco constantemente.

---

# Dashboard

Indicadores devem ser derivados de eventos.

Nunca depender apenas de consultas complexas.

---

# Auditoria

Eventos administrativos sempre geram registro.

Exemplo:

admin.company_created

admin.permission_changed

---

# Falhas

Eventos críticos devem possuir:

retry;

dead letter;

alerta.

---

# Dead Letter Queue

Eventos que falharam após tentativas devem ser armazenados para análise.

---

# Evolução

Novos eventos devem:

ser documentados;

possuir versão;

não quebrar consumidores existentes.

---

# Regra Final

Eventos são a linguagem comum do sistema.

O Conecta Flow deve evoluir adicionando novos consumidores, não criando dependências frágeis.
