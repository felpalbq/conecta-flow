# AI Development Principles
Version: 1.0

---

# Objetivo

Este documento define como agentes de IA devem desenvolver o Conecta Flow.

Não descreve tecnologias.

Não descreve funcionalidades.

Ele define princípios obrigatórios.

Toda implementação deve respeitar estes princípios antes de escrever qualquer linha de código.

---

# Filosofia

O objetivo não é produzir código rapidamente.

O objetivo é construir uma plataforma que permaneça organizada durante anos.

Sempre priorizar qualidade sobre velocidade.

---

# Ordem de Desenvolvimento

Sempre seguir esta sequência.

1.

Compreender o domínio.

↓

2.

Consultar documentação existente.

↓

3.

Planejar a alteração.

↓

4.

Somente então escrever código.

Nunca inverter essa ordem.

---

# Nunca Assumir

Caso exista dúvida.

Nunca assumir comportamento.

Consultar documentação.

Caso a documentação não exista.

Criar documentação primeiro.

Somente depois implementar.

---

# Documentação é Código

Toda decisão arquitetural importante deve ser documentada.

Nunca implementar funcionalidades relevantes apenas através do código.

---

# Conversation First

Todo desenvolvimento deve responder.

Como esta alteração melhora uma Conversation?

Caso não exista resposta clara.

A alteração provavelmente não pertence ao produto.

---

# Core First

Nunca alterar módulos antes do Core.

Nunca criar dependências do Core para módulos opcionais.

---

# Event First

Sempre pensar em eventos antes de pensar em tabelas.

Pergunta correta.

Qual evento aconteceu?

Não.

Qual tabela será atualizada?

---

# Domain First

Toda implementação começa pelo domínio.

Nunca pelo banco.

Nunca pelo frontend.

---

# Banco

O banco implementa o domínio.

Nunca o contrário.

---

# IA

A IA nunca é responsável pelas regras do negócio.

Ela apenas interpreta.

Toda decisão pertence ao domínio.

---

# n8n

O n8n nunca contém regras centrais.

Ele apenas orquestra processos.

---

# Frontend

O frontend representa estado.

Nunca cria regras de negócio.

---

# Segurança

Toda funcionalidade deve responder.

Quem pode acessar?

Quem pode alterar?

Como será auditada?

---

# Multi-tenancy

Toda alteração deve preservar isolamento entre empresas.

Nunca criar exceções.

---

# Simplicidade

Sempre escolher a solução mais simples.

Evitar abstrações prematuras.

---

# Componentes

Criar componentes pequenos.

Cada componente possui apenas uma responsabilidade.

---

# Funções

Funções pequenas.

Responsabilidade única.

Nome explícito.

---

# Eventos

Sempre publicar eventos relevantes.

Nunca criar dependências ocultas.

---

# Serviços

Toda lógica reutilizável pertence a serviços.

Nunca duplicar regras.

---

# Banco

Nunca escrever SQL espalhado pelo projeto.

Sempre centralizar acesso.

---

# Infraestrutura

Toda integração externa passa por adaptadores.

Nunca espalhar chamadas HTTP.

---

# Refatoração

Sempre melhorar código existente antes de duplicar lógica.

---

# Performance

Otimizar apenas após existir necessidade comprovada.

Nunca otimizar prematuramente.

---

# Testes

Toda regra importante deve ser testável.

Arquitetura deve favorecer testes.

---

# Comentários

Escrever código que dispense comentários.

Comentários apenas quando agregam contexto.

Nunca explicar código óbvio.

---

# Dependências

Adicionar bibliotecas apenas quando realmente agregam valor.

Preferir código simples.

---

# Git

Commits pequenos.

Responsabilidade única.

Mensagens claras.

---

# Qualidade

Antes de concluir qualquer tarefa perguntar.

O domínio continua consistente?

Existe acoplamento?

Existe duplicação?

Existe solução mais simples?

---

# Regra Final

O Conecta Flow deve crescer preservando sua arquitetura.

Nenhuma implementação vale a perda de consistência arquitetural.
