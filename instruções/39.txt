# Supabase Architecture
Version: 1.0

---

# Objetivo

Definir como o Supabase será utilizado como infraestrutura principal do Conecta Flow.

Este documento estabelece:

- responsabilidades;
- banco;
- autenticação;
- segurança;
- storage;
- realtime;
- migrations;
- integração com desenvolvimento.

---

# Papel do Supabase

O Supabase será responsável por:

- PostgreSQL;
- autenticação;
- autorização;
- armazenamento;
- realtime;
- funções server-side.

---

# Não Responsabilidades

O Supabase não deve conter:

- regras complexas de negócio;
- decisões da IA;
- workflows de automação;
- lógica comercial.

---

# Arquitetura Geral

```

Application

↓

Supabase Client

↓

Supabase Services

|

├── Auth

├── PostgreSQL

├── Storage

├── Realtime

└── Edge Functions

```

---

# Banco de Dados

O banco principal será:

PostgreSQL.

---

# Princípio

O banco representa o domínio.

Não apenas armazenamento.

---

# Multi-Tenancy

Todas as entidades empresariais possuem:

```
company_id
```

---

Exemplos:

contacts

conversations

messages

agents

documents

integrations

---

# RLS

Row Level Security é obrigatório.

---

Nenhuma tabela de negócio deve existir sem:

- política RLS;
- teste de isolamento.

---

# Exemplo

Tabela:

conversations

---

Regra:

Usuário só pode visualizar conversas da própria empresa.

---

# Auth

Responsabilidade:

Supabase Auth.

---

Inicialmente:

- email;
- senha;
- recuperação.

---

Futuro:

- OAuth;
- magic link;
- autenticação adicional.

---

# Relacionamento Auth x Usuário

Supabase Auth:

identidade.

Tabela própria:

profiles/users.

---

Modelo:

```

auth.users

↓

profiles

↓

company_memberships

↓

companies

```

---

# Companies

Tabela principal de tenants.

---

Responsabilidades:

- identificação;
- status;
- plano;
- configurações.

---

# Company Memberships

Relacionamento:

Usuário

+

Empresa.

---

Campos conceituais:

```

user_id

company_id

role

permissions

status

```

---

# Permissões

Não confiar apenas em frontend.

---

Validação:

Frontend

+

Backend

+

RLS

---

# Migrations

Toda alteração estrutural deve usar migration.

---

Nunca:

alterar produção manualmente.

---

Estrutura:

```

supabase/

migrations/

```

---

# Ordem das migrations

Inicial:

1.

Extensions


2.

Companies


3.

Profiles


4.

Memberships


5.

Core entities


6.

Policies


---

# Seed

Utilizado apenas para:

- desenvolvimento;
- testes.

---

Nunca inserir:

dados reais.

---

# Storage

Responsável por:

- arquivos;
- documentos;
- mídias.

---

Exemplos:

- anexos de conversa;
- documentos da empresa;
- base da IA.

---

# Estrutura lógica

```

storage/

company_id/

   conversations/

   knowledge/

   uploads/

```

---

# Segurança Storage

Arquivos privados.

---

Nunca utilizar:

URLs públicas permanentes.

---

# Realtime

Utilizado para:

atualizações instantâneas.

---

Primeiro uso:

Inbox.

---

Exemplo:

Cliente envia mensagem.

↓

Evento salvo.

↓

Realtime atualiza atendente.

---

# Não usar Realtime para:

processos complexos.

---

# Edge Functions

Usar quando:

- precisa executar servidor seguro;
- proteger segredo;
- criar endpoint específico.

---

Exemplos:

- webhook;
- processamento seguro;
- validação externa.

---

# Não colocar:

toda regra do sistema.

---

# Database Functions

Utilizar apenas quando:

- operação é naturalmente do banco;
- precisa consistência transacional.

---

# Índices

Todas tabelas multi-tenant devem considerar:

```
company_id
```

---

Tabelas críticas:

conversations

messages

contacts

events

---

# Auditoria

Eventos administrativos devem ser persistidos.

---

Tabela conceitual:

audit_logs

---

Campos:

```

actor

company_id

action

entity

timestamp

metadata

```

---

# Eventos

Tabela:

events

---

Responsável por:

histórico;

processamento;

auditoria.

---

# Vetores / RAG

Inicialmente:

Postgres + extensão vetorial.

---

Separação:

```

knowledge_documents

knowledge_chunks

embeddings

```

---

Sempre:

company_id.

---

# MCP Supabase

O Claude Code poderá utilizar MCP.

---

Objetivos:

- consultar estrutura;
- criar migrations;
- validar dados;
- auxiliar desenvolvimento.

---

# Segurança MCP

Credenciais serão adicionadas manualmente.

---

Nunca armazenar:

tokens.

---

# Ambientes

Separar:

Development

Staging

Production

---

Cada ambiente possui:

- banco;
- credenciais;
- storage;
- configurações próprias.

---

# Backup

Produção deve possuir:

backup;

recuperação;

monitoramento.

---

1.

2.

Configurar Auth.


3.

Criar migrations iniciais.


4.

Criar RLS.


5.

Gerar tipos.


6.

Conectar aplicação.


---

# Checklist

Antes de usar banco:

[x] Multi-tenancy definido

[x] RLS planejado

[x] Auth definido

[x] Migrations configuradas

[x] Storage protegido

[x] Ambientes separados

---

# Regra Final

Supabase deve proteger o domínio.

Nunca ser apenas um local onde dados são guardados.
