# Event Architecture
Version: 1.0

---

# Objetivo

Definir a arquitetura orientada a eventos do Conecta Flow.

Toda alteração relevante dentro da plataforma deverá gerar um evento.

Eventos representam fatos ocorridos.

Eles nunca representam intenções.

---

# Filosofia

O sistema é Event Driven.

A plataforma evolui através de acontecimentos.

Nunca através de chamadas diretas entre módulos.

Sempre que possível:

Acontecimento

↓

Evento

↓

Consumidores

↓

Novos acontecimentos

Essa arquitetura reduz acoplamento e facilita expansão do produto.

---

# O que é um Evento

Um evento representa algo que aconteceu.

Exemplos.

Mensagem recebida.

Mensagem enviada.

Usuário assumiu atendimento.

Lead qualificado.

Pagamento confirmado.

Agendamento criado.

Nunca.

"Qualificar Lead"

Isso é um comando.

Evento sempre representa passado.

# Ordem

Eventos nunca assumem ordem fixa.

Todo consumidor deve ser resiliente.

Nunca depender que outro consumidor terminou antes.

---

# Idempotência

Todo consumidor deve conseguir processar o mesmo evento mais de uma vez.

Sem efeitos colaterais.

Duplicidade nunca deve gerar inconsistência.

---

# Histórico

Eventos nunca são alterados.

Nunca apagados.

Nunca reescritos.

Caso algo mude.

Um novo evento deve ser criado.

---

# Auditoria

Toda auditoria será baseada em eventos.

Nunca em estados.

Isso permite reconstruir toda operação.

---

# Dashboard

Os indicadores deverão ser calculados utilizando eventos.

Nunca consultas específicas para cada tela.

---

# IA

A IA reage aos eventos.

Nunca monitora tabelas diretamente.

Exemplo.

Message Received

↓

Workflow IA

↓

Resposta

↓

Message Sent

↓

Novo Evento

---

# n8n

O n8n reage aos eventos.

Nunca consulta constantemente o banco procurando alterações.

---

# Mission Control

Mission Control poderá acompanhar eventos em tempo real.

Sem acessar diretamente módulos internos.

---

# Módulos

Novos módulos apenas criam novos consumidores.

Nunca modificam eventos existentes.

Isso garante compatibilidade.

---

# Versionamento

Eventos poderão evoluir.

Sempre mantendo versão.

Nunca alterar o significado de um evento existente.

---

# Regra Fundamental

Todo fato importante gera um evento.

Todo evento pode possuir múltiplos consumidores.

Nenhum consumidor conhece os demais.

Essa é a principal estratégia para manter o sistema desacoplado.
