# N8N Guidelines
Version: 1.0

---

# Objetivo

Definir o papel do n8n dentro da arquitetura do Conecta Flow.

O n8n é um orquestrador de eventos.

Nunca é o backend.

Nunca contém regras centrais do negócio.

---

# Filosofia

A aplicação decide.

O n8n executa.

A aplicação conhece o domínio.

O n8n conhece processos.

---

# Responsabilidades

O n8n é responsável por:

- integrar sistemas
- reagir a eventos
- executar tarefas assíncronas
- controlar filas
- enviar notificações
- sincronizar dados
- executar workflows
- automatizar processos

Nunca substituir a aplicação.

---

# Arquitetura

Evento

↓

Application

↓

Webhook

↓

n8n

↓

Integrações

↓

Resultado

↓

Application

Toda alteração definitiva retorna para a aplicação.

---

# Nunca fazer no n8n

Nunca implementar:

regras comerciais

autenticação

autorização

controle de permissões

lead score

RAG

memória da IA

persistência principal

consultas SQL complexas

lógica de planos

lógica de módulos

---

# Sempre fazer no n8n

Chamadas HTTP

Integrações

Webhooks

Cron Jobs

Notificações

Filas

Transformações simples

Retentativas

Execuções assíncronas

---

# Organização

Cada workflow possui apenas uma responsabilidade.

Nunca criar workflows gigantes.

---

# Estrutura

Trigger

↓

Validação

↓

Processamento

↓

Integração

↓

Resposta

↓

# Nome dos Workflows

Formato obrigatório.

categoria.descrição

Exemplos

conversation.receive-message

conversation.send-message

conversation.followup

lead.qualification

appointment.reminder

payment.webhook

instagram.receive-message

system.cleanup

---

# Pastas

Organizar por domínio.

Conversation

Lead

Agent

Campaigns

Payments

Appointments

Infrastructure

Mission Control

---

# Entrada

Toda entrada deve ser validada.

Nunca confiar em payloads externos.

---

# Saída

Toda saída deve possuir:

status

timestamp

correlation_id

empresa

origem

resultado

---

# Erros

Todo erro deve:

registrar log

notificar quando crítico

permitir retentativa

Nunca falhar silenciosamente.

---

# Retentativas

Implementar retry apenas em:

timeout

erro temporário

rate limit

Nunca repetir operações destrutivas automaticamente.

---

# Idempotência

Todo workflow deve poder ser executado novamente.

Sem gerar efeitos duplicados.

---

# Logs

Registrar:

workflow

empresa

evento

tempo

resultado

erro

Nunca registrar informações sensíveis.

---

# Segredos

Utilizar Credentials do n8n.

Nunca hardcode.

Nunca salvar tokens em nós.

---

# Variáveis

Utilizar Environment Variables.

Nunca repetir valores.

---

# Banco

O n8n nunca acessa tabelas diretamente quando existir API disponível.

Prioridade:

Application API

↓

Edge Functions

↓

RPC

↓

SQL (último recurso)

---

# IA

O n8n nunca conversa diretamente com o modelo.

Sempre utilizar o serviço do Conversation Agent.

---

# WhatsApp

Recebe mensagens.

Encaminha para a aplicação.

Recebe resposta.

Envia ao provedor.

Nunca interpreta mensagens.

---

# Evolution API

Enquanto utilizada:

Receber eventos

Enviar mensagens

Monitorar conexão

Nunca implementar regras de negócio.

---

# Meta API

Mesmo comportamento.

Apenas troca o adaptador.

---

# Agendamento

Workflow apenas executa.

A lógica de disponibilidade pertence ao domínio.

---

# Campanhas

Workflow dispara.

Segmentação pertence ao backend.

---

# Pagamentos

Workflow recebe webhook.

Validação pertence à aplicação.

---

# Observabilidade

Cada workflow deve registrar:

início

fim

duração

resultado

falhas

---

# Versionamento

Todo workflow deve possuir:

nome

versão

data

autor

histórico

---

# Backup

Exportar workflows periodicamente.

Versionar no Git.

Nunca manter apenas dentro do n8n.

---

# Desenvolvimento

Todo workflow deve possuir documentação correspondente.

Nunca criar workflows "temporários".

---

# Regra Fundamental

Se um workflow desaparecer hoje, a aplicação continua consistente.

O n8n acelera processos.

Nunca define o comportamento do sistema.
