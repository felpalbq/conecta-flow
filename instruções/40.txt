# n8n Automation Architecture
Version: 1.0

---

# Objetivo

Definir o papel do n8n dentro da arquitetura do Conecta Flow.

Este documento estabelece:

- responsabilidades;
- limites;
- padrões de workflow;
- integrações;
- tratamento de erros.

---

# Filosofia

n8n é uma camada de automação.

Não é o núcleo da aplicação.

---

# Responsabilidade do n8n

O n8n deve cuidar de:

- comunicação entre sistemas;
- eventos externos;
- tarefas automáticas;
- processos assíncronos;
- integrações.

---

# Não Responsabilidade

O n8n não deve conter:

- regras principais do negócio;
- permissões;
- lógica de domínio;
- decisões comerciais.

---

# Arquitetura

```

External Services

       |

      n8n

       |

Application Services

       |

     Supabase


```

---

# Exemplos de uso correto

Mensagem recebida WhatsApp.

↓

Webhook n8n

↓

Criar evento

↓

Aplicação processa conversa.

---

# Exemplo incorreto

Webhook WhatsApp

↓

n8n decide:

"Esse cliente é lead quente"

↓

n8n altera banco.

---

A decisão pertence ao domínio.

---

# Tipos de Workflow

Os workflows serão divididos em categorias.

---

# 1. Incoming Events

Eventos externos entrando no sistema.

Exemplos:

- mensagem recebida;
- pagamento confirmado;
- formulário enviado.

---

Padrão:

```
WF_SOURCE_EVENT_VERSION
```

Exemplo:

```
WF_WHATSAPP_MESSAGE_RECEIVED_V1
```

---

# 2. Background Jobs

Processos automáticos.

Exemplos:

- follow-up;
- lembretes;
- sincronizações.

---

Exemplo:

```
WF_FOLLOWUP_EXECUTION_V1
```

---

# 3. Notifications

Comunicação interna.

Exemplos:

- avisar atendente;
- alertar erro;
- enviar resumo.

---

# 4. Integrations

Conectores externos.

Exemplo:

- Google Calendar;
- Meta;
- pagamentos.

---

# Fluxo padrão

Todo workflow deve seguir:

```

Trigger

↓

Validation

↓

Processing

↓

Application Action

↓

Logging

↓

Response

```

---

# Webhooks

Todo webhook deve:

1.

Validar origem.


2.

Validar assinatura quando existir.


3.

Registrar evento.


4.

Responder rapidamente.


---

# Eventos

O n8n não deve manipular diretamente tabelas críticas.

---

Preferência:

n8n cria evento.

Aplicação processa.

---

Exemplo:

```

message.received

conversation.created

lead.updated

```

---

# Tratamento de Erros

Todo workflow deve possuir:

- captura de erro;
- registro;
- tentativa novamente;
- alerta.

---

# Nunca

Workflow silenciosamente falhar.

---

# Retry

Definir:

- número de tentativas;
- intervalo;
- ação final.

---

# Logs

Todo workflow deve registrar:

- início;
- fim;
- status;
- erro;
- payload resumido.

---

# Dados Sensíveis

Nunca armazenar:

- tokens;
- senhas;
- dados pessoais desnecessários.

---

# Credenciais

Devem ficar:

n8n Credentials Manager.

---

Nunca:

- dentro do JSON;
- no GitHub;
- em documentação.

---

# WhatsApp

Inicialmente:

Evolution API.

Objetivo:

validar produto.

---

Produção:

Meta Official API.

---

Responsabilidade n8n:

- receber webhook;
- enviar mensagens;
- sincronizar eventos.

---

Não responsável:

- interpretar conversa.

---

```

Scheduler

↓

Buscar contatos elegíveis

↓

Enviar evento

↓

Agente decide ação

```

---

Fluxo:

```

Segmentação

↓

Aprovação

↓

Disparo

↓

Monitoramento

```

---

# Agendamento Futuro

Fluxo:

```

Solicitação

↓

Disponibilidade

↓

Confirmação

↓

Lembrete

```

---

# Integração com IA

n8n pode:

- chamar serviços;
- iniciar processamento;
- enviar contexto.

---

Mas:

decisão do agente pertence ao módulo AI.

---

# Ambiente Local

Inicialmente:

Docker.

---

Estrutura:

```

n8n/

docker-compose.yml

workflows/

credentials/

```

---

# Produção

Futuramente avaliar:

- n8n cloud;
- servidor dedicado;
- container próprio.

---

# Versionamento

Workflows exportados devem ser versionados.

---

Exemplo:

```

n8n/workflows/

WF_WHATSAPP_MESSAGE_RECEIVED_V1.json

```

---

# Alterações

Nova versão:

```

V1

V2

V3

```

---

Não sobrescrever histórico.

---

# Segurança

n8n deve possuir:

- autenticação;
- acesso restrito;
- backups.

---

# Monitoramento

Acompanhar:

- execuções;
- falhas;
- tempo médio;
- consumo.

---

# Relação com Mission Control

Mission Control acompanha:

- saúde;
- erros;
- consumo.

---

Não administra workflows diariamente.

---

# Critério de qualidade

Um workflow deve ser:

- simples;
- rastreável;
- substituível.

---

# Regra Final

n8n conecta o sistema.

Não define o sistema.
