# Security Model
Version: 1.0

---

# Objetivo

Definir os princípios e mecanismos de segurança do Conecta Flow.

Toda implementação deve respeitar este documento.

Segurança não deve ser adicionada posteriormente.

---

# Filosofia

O sistema deve assumir que:

- usuários podem cometer erros;
- integrações podem falhar;
- credenciais podem vazar;
- dados são sensíveis;
- toda ação deve ser rastreável.

A arquitetura deve limitar impactos.

---

# Princípios

## Least Privilege

Todo usuário possui apenas as permissões necessárias.

Nunca conceder acesso amplo por conveniência.

---

## Defense in Depth

A segurança deve existir em múltiplas camadas.

Exemplo:

Frontend

↓

API

↓

Domínio

↓

Banco

↓

RLS

---

## Zero Trust

Nenhuma requisição é confiável automaticamente.

Toda ação deve validar:

Quem?

Qual empresa?

Qual permissão?

Qual contexto?

---

# Multi-Tenancy

A plataforma é multi-tenant desde o início.

Toda informação pertence a uma empresa.

Nunca permitir acesso cruzado.

---

# Isolamento

Toda tabela operacional deve possuir:

company_id

Toda consulta deve considerar:

company_id

---

# Supabase RLS

Row Level Security é obrigatório.

Nenhuma tabela de negócio deve existir sem política de acesso.

---

# Regras RLS

Usuário comum:

Pode acessar somente sua empresa.

---

Administrador:

Pode acessar múltiplas empresas conforme permissão.

---

Sistema:

Utiliza service role somente em operações controladas.

---

# Nunca

Nunca confiar em:

- filtro do frontend;
- parâmetro enviado pelo usuário;
- ID informado na URL.

O banco deve garantir isolamento.

---

# Usuários

Existem quatro atores principais.

---

# Cliente

Pessoa que conversa pelo canal.

Não possui acesso ao sistema.

---

# Atendente

Usuário operacional.

Pode:

- visualizar conversas permitidas;
- responder;
- assumir atendimento;
- utilizar agenda quando disponível;
- consultar contatos.

Não pode:

- alterar configurações críticas;
- acessar outras empresas;
- administrar usuários.

---

# Agente Inteligente

Não possui usuário tradicional.

Opera através de serviços internos.

Nunca recebe permissões amplas.

---

# Administrador

Mission Control.

Possui acesso administrativo.

Todas as ações são auditadas.

---

# Autenticação

Responsabilidade:

Supabase Auth.

---

Deve suportar:

login

sessão

recuperação

expiração

revogação

---

# Autorização

Nunca baseada apenas no frontend.

Sempre validar no backend.

---

# Modelo de Permissões

Permissões granulares.

Exemplo:

conversation.read

conversation.reply

conversation.assign

contact.edit

appointment.create

campaign.execute

---

# Proibido

Permissões genéricas:

admin

manager

superuser

quando não forem realmente necessárias.

---

# Handoff

Toda mudança de responsável deve gerar evento.

Registrar:

usuário anterior

novo usuário

motivo

timestamp

---

# Auditoria

Toda ação administrativa deve gerar log.

Exemplos:

Criar usuário.

Alterar permissão.

Ativar módulo.

Alterar prompt.

Alterar integração.

---

# Audit Log

Nunca pode ser apagado.

Nunca pode ser alterado.

---

# Dados Sensíveis

Considerar sensíveis:

mensagens

documentos

telefones

emails

tokens

credenciais

conhecimento da empresa

---

# Armazenamento

Nunca armazenar:

senhas

tokens em texto puro

chaves privadas

segredos

---

# Secrets

Utilizar:

Environment Variables

Secret Managers

Credentials do n8n

Nunca:

Código

Git

Banco

---

# IA

A IA possui regras especiais.

---

# Context Isolation

O contexto enviado ao modelo deve pertencer exclusivamente à empresa.

Nunca misturar:

empresas

clientes

bases vetoriais

memórias

---

# RAG

Cada empresa possui sua própria separação lógica.

Exemplo:

company_id

metadata

namespace

---

# Prompt Security

Prompts possuem:

versão

autor

data

histórico

---

# Tool Security

Ferramentas da IA possuem permissões próprias.

A IA não pode:

executar ações administrativas;

acessar dados fora do contexto;

alterar configurações.

---

# Uploads

Todo arquivo deve validar:

tipo

tamanho

origem

permissão

---

# Storage

Arquivos privados.

Nunca utilizar URLs públicas permanentes.

---

# Integrações

Toda integração deve armazenar:

status

última sincronização

erro

credencial válida

---

# Webhooks

Toda entrada externa deve validar:

assinatura

origem

payload

timestamp

---

# WhatsApp

Mensagens devem validar:

empresa

canal

conversation

provider id

---

# Logs

Nunca registrar:

senhas

tokens

conteúdo privado completo

dados financeiros sensíveis

---

# Rate Limit

Aplicar limites em:

login

APIs públicas

webhooks

IA

envios

---

# Backup

Backups devem preservar:

isolamento

integridade

criptografia

controle de acesso

---

# Desenvolvimento

Ambientes separados:

Development

Staging

Production

Nunca utilizar dados reais em desenvolvimento.

---

# Testes

Testar obrigatoriamente:

isolamento entre empresas;

permissões;

acesso negado;

expiração de sessão;

ações administrativas.

---

# Regra Final

Nenhuma funcionalidade está pronta até responder:

Quem pode usar?

Quem pode ver?

Quem pode alterar?

Como será auditado?

Como evitar vazamento?
