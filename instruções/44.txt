# Multi Tenant Integration Architecture
Version: 1.0

---

# Objetivo

Definir como o Conecta Flow gerencia integrações externas em uma arquitetura SaaS multi-tenant.

Este documento estabelece:

- isolamento por empresa;
- gerenciamento de providers;
- arquitetura de adapters;
- funcionamento dos webhooks;
- relação entre n8n, aplicação e integrações.

---

# Princípio Fundamental

Cada empresa possui suas próprias integrações.

Porém:

A plataforma possui uma arquitetura única.

---

# Regra principal

Personalização por configuração.

Não por código.

---

# Modelo incorreto

Não criar:

```
Empresa A
    |
    Workflow próprio
    |
    Código próprio


Empresa B
    |
    Workflow próprio
    |
    Código próprio
```

---

Problema:

- baixa escalabilidade;
- difícil manutenção;
- divergência entre clientes;
- ausência de produto SaaS real.

---

# Modelo correto

```
                 Plataforma

                     |

          Integration Layer

                     |

        Configuração da Empresa

                     |

       Provider específico


```

---

# Empresa

Cada empresa possui:

- cadastro;
- usuários;
- configurações;
- integrações;
- credenciais;
- regras próprias.

# Provider Layer

O sistema não deve conhecer diretamente fornecedores.

---

Arquitetura:

```

WhatsApp Domain

        |

Provider Interface

        |

---------------------

|          |          |

Meta    Evolution   ZAPI


```

---

# Interface comum

Todos providers devem entregar:

```
NormalizedMessage

NormalizedContact

MessageStatus

```

---

# Exemplo

Entrada Evolution:

```
{
sender,
message,
timestamp
}

```

---

Entrada Meta:

```
{
contacts,
messages,
metadata
}

```

---

Saída interna:

```
{
company_id,

phone,

message,

media,

timestamp,

provider

}

```

---

# Webhooks

Cada empresa possui um endpoint ou identificação própria.

---

Exemplo:

```
/webhooks/whatsapp/{company_token}

```

---

Fluxo:

```

Mensagem recebida

↓

Provider

↓

Webhook

↓

Identificação empresa

↓

Normalização

↓

Evento interno

↓

Processamento

```

---

# n8n

Responsabilidade:

Orquestração.

---

O n8n:

- recebe eventos externos;
- chama APIs;
- executa automações;
- conecta serviços.

---

O n8n NÃO:

- decide regras comerciais;
- classifica leads sozinho;
- controla permissões;
- substitui backend.

---

# Workflow Multi Tenant

O padrão:

```
WF_WHATSAPP_MESSAGE_RECEIVED_V1

```

atende todas empresas.

---

Fluxo:

```

Webhook

↓

Identifica company_id

↓

Busca configuração

↓

Seleciona provider

↓

Normaliza evento

↓

Envia aplicação

```

---

# Configuração por empresa

Cada empresa pode possuir:

- provider diferente;
- mensagens padrão diferentes;
- regras de handoff diferentes;
- horários diferentes;
- comportamento de IA diferente.

---

# Exemplo

Empresa A:

```
IA responde automaticamente.

Handoff após orçamento.

```

Empresa B:

```
IA apenas triagem.

Humano assume cedo.

```

---

# Credenciais

Nunca armazenar:

- tokens;
- chaves;
- senhas;

em:

- código;
- GitHub;
- documentação.

---

Utilizar:

- secrets;
- environment variables;
- credential manager.

---

# Mission Control

Administrador deve visualizar:

- integrações cadastradas;
- provider;
- status;
- erros;
- consumo.

---

Não deve editar código.

---

# Clientes Enterprise

Exceção:

Quando uma empresa exige integração exclusiva.

---

Pode existir:

```
WF_CUSTOMER_X

```

ou:

```
CUSTOM_ADAPTER_X

```

---

Mas isso é exceção.

---

# Escalabilidade

A arquitetura deve suportar:

10 empresas

100 empresas

1000 empresas

sem criar uma aplicação por cliente.

---

# Regra Final

O Conecta Flow é uma plataforma multi-tenant.

Empresas possuem configurações diferentes.

Não sistemas diferentes.
