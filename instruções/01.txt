# Product Principles
Version: 1.0

---

# Objetivo

Este documento define os princípios que orientam todas as decisões de desenvolvimento do Conecta Flow.

Sempre que existir dúvida entre duas abordagens técnicas, arquiteturais ou de experiência do usuário, este documento deve prevalecer.

O objetivo não é criar o software mais completo.

O objetivo é criar o software mais útil para pequenas empresas que utilizam o WhatsApp como principal canal de atendimento.

---

# Princípio 1

## A conversa é o centro do sistema.

O Conecta Flow não é um CRM tradicional.

Toda informação do sistema deve existir para enriquecer uma conversa.

Não existem módulos independentes competindo pela atenção do usuário.

Tudo converge para a conversa.

Exemplos:

- Lead Score existe para ajudar durante a conversa.
- Agenda existe para dar continuidade à conversa.
- Pagamento existe para concluir uma conversa.
- Dashboard existe para mostrar o resultado das conversas.

---

# Princípio 2

## O usuário trabalha na conversa.

O operador nunca deve precisar navegar entre diversas telas para concluir um atendimento.

Durante uma conversa ele deve possuir acesso ao contexto completo.

A conversa deve concentrar:

- mensagens;
- histórico;
- informações do contato;
- score;
- observações;
- arquivos;
- agendamentos;
- follow-ups;
- ações rápidas.

Sempre que possível, evitar abrir novas páginas.

---

# Princípio 3

## A interface deve ser imediatamente familiar.

O usuário já sabe utilizar o WhatsApp Business.

O sistema deve preservar essa familiaridade.

Sempre utilizar padrões conhecidos quando possível.

Exemplos:

- layout semelhante;
- comandos "/";
- respostas rápidas;
- anexos;
- emojis;
- atalhos de teclado;
- comportamento esperado.

Nunca reinventar uma interface consolidada sem um motivo extremamente forte.

---

# Princípio 4

## IA é uma colaboradora.

A Inteligência Artificial faz parte da equipe.

Ela possui responsabilidades próprias.

Ela pode:

- responder;
- qualificar;
- organizar;
- resumir;
- acompanhar;
- recuperar oportunidades;
- executar tarefas.

Ela não substitui completamente o humano.

Quando necessário deve solicitar handoff.

---

# Princípio 5

## O humano assume quando agrega valor.

A plataforma nunca deve obrigar a IA a resolver tudo.

Da mesma forma, nunca deve obrigar o humano a assumir tarefas repetitivas.

O sistema deve encontrar automaticamente o melhor equilíbrio entre automação e atendimento humano.

---

# Princípio 6

## IA custa dinheiro.

Cada chamada para um modelo possui custo.

Cada processamento possui impacto financeiro.

Portanto:

- evitar chamadas desnecessárias;
- utilizar regras simples quando possível;
- reutilizar contexto;
- evitar processamento duplicado;
- preferir decisões determinísticas antes da LLM.

A IA deve pensar apenas quando necessário.

---

# Princípio 7

## Primeiro filtrar.

Nem toda conversa merece processamento inteligente.

Antes da qualificação existe uma etapa obrigatória de triagem.

Objetivos:

- eliminar spam;
- eliminar enganos;
- separar contatos pessoais;
- identificar fornecedores;
- identificar funcionários;
- reduzir processamento.

Somente após essa etapa a conversa pode se tornar um lead.

---

# Princípio 8

## Lead não nasce automaticamente.

Uma conversa não é um lead.

Ela torna-se um lead quando demonstra potencial comercial.

Até esse momento ela permanece apenas como uma conversa.

Essa distinção reduz complexidade e melhora a qualidade das métricas.

---

# Princípio 9

## O dashboard demonstra valor.

O Dashboard não existe para análise profunda.

Sua função é mostrar:

- economia;
- produtividade;
- resultados;
- oportunidades;
- atividades recentes.

O usuário deve compreender rapidamente o impacto da plataforma.

---

# Princípio 10

## O Core deve permanecer pequeno.

Toda funcionalidade nova deve responder:

Ela fortalece o Core?

Ou pertence a um módulo?

Sempre que possível optar por módulos independentes.

---

# Princípio 11

## O sistema cresce por capacidades.

Novas funcionalidades devem ser adicionadas como capacidades plugáveis.

Exemplos:

Core

↓

Agenda

↓

Campanhas

↓

Pagamento

↓

Instagram

↓

Landing Pages

↓

Google Calendar

↓

Novos canais

Nunca criar versões diferentes do sistema.

Existe apenas um produto.

Os módulos apenas expandem suas capacidades.

---

# Princípio 12

## Multiempresa desde o primeiro dia.

Toda decisão técnica deve considerar:

- isolamento completo dos dados;
- segurança;
- permissões;
- escalabilidade;
- configuração independente.

Nenhum recurso poderá assumir que existe apenas uma empresa.

---

# Princípio 13

## O cliente administra o negócio.

Não administra tecnologia.

O usuário nunca deve precisar configurar:

- banco vetorial;
- prompts;
- workflows;
- integrações;
- modelos;
- banco de dados.

Esses elementos pertencem ao Mission Control.

---

# Princípio 14

## O Mission Control é outro produto.

O painel administrativo não faz parte da experiência do cliente.

Ele possui objetivos completamente diferentes.

Seu foco é:

- operação;
- monitoramento;
- suporte;
- evolução;
- segurança;
- custos.

Nunca misturar funcionalidades administrativas com o ambiente do cliente.

---

# Princípio 15

## Segurança faz parte da arquitetura.

Segurança não será adicionada posteriormente.

Ela deve existir desde a primeira linha de código.

Toda funcionalidade deve respeitar:

- autenticação;
- autorização;
- isolamento;
- auditoria;
- rastreabilidade;
- menor privilégio possível.

---

# Princípio 16

## Toda decisão deve reduzir complexidade.

Antes de adicionar qualquer recurso perguntar:

- Existe uma solução mais simples?

- O usuário realmente precisa disso?

- Isso reduz trabalho?

- Isso melhora o atendimento?

Caso contrário, a funcionalidade provavelmente não pertence ao produto.

---

# Regra de Ouro

Toda funcionalidade deve responder positivamente pelo menos uma destas perguntas.

1. Melhora uma conversa?

2. Melhora o resultado da conversa?

3. Melhora a operação da plataforma?

Se a resposta for negativa para todas, a funcionalidade não deve ser implementada.
