# Multi Tenancy Model
Version: 1.0

---

# Objetivo

Definir como o Conecta Flow suporta múltiplas empresas dentro da mesma plataforma.

Este documento define:

- isolamento;
- propriedade dos dados;
- acesso;
- criação de tenants;
- relacionamento entre usuários e empresas;
- comportamento administrativo.

---

# Filosofia

Cada empresa é um ambiente isolado dentro da plataforma.

Compartilham:

- infraestrutura;
- aplicação;
- código;
- serviços.

Nunca compartilham:

- dados;
- contexto;
- conhecimento;
- conversas;
- configurações.

---

# Conceito

Tenant = Empresa cliente.

Cada tenant possui:

- usuários;
- contatos;
- conversas;
- mensagens;
- configurações;
- integrações;
- módulos;
- conhecimento da IA.

---

```
Platform

│      ├── Contacts

│      ├── Conversations

│      ├── AI Knowledge

│      └── Integrations


└── Mission Control

```

# Regra Fundamental

Nenhuma consulta de dados empresariais pode existir sem contexto de empresa.

---

# Contexto do Tenant

Toda requisição deve carregar:

user_id

company_id

permissions

session

---

# Nunca confiar

Nunca utilizar:

company_id enviado pelo frontend.

O frontend informa intenção.

O backend valida autoridade.

# Dados do Cliente Final

Pertencem sempre à empresa.

Um contato WhatsApp recebido por uma clínica pertence à clínica.

Nunca pertence à plataforma.

# IA

A separação da IA é obrigatória.

Cada empresa possui:

- prompts;
- políticas;
- memória;
- documentos;
- embeddings;
- configurações.

# Integrações

Cada empresa possui suas próprias credenciais.

Nunca compartilhar tokens.

---

# WhatsApp

Uma empresa:

um número.

ou

múltiplos números.

A arquitetura deve permitir ambos.

---

# Storage

Arquivos devem possuir isolamento lógico.

Exemplo:

```

company_id/

    conversations/

    documents/

    uploads/

```

---

# Logs

Todo log operacional deve possuir:

company_id

quando aplicável.

---

# Mission Control

É a única área que pode atravessar tenants.

Mesmo assim:

toda ação deve ser auditada.

---

# Administrador

Pode:

visualizar empresas;

diagnosticar problemas;

atuar tecnicamente.


Não deve:

alterar dados comerciais sem registro.

---

# Exportação

Toda empresa deve poder futuramente:

exportar seus dados;

receber histórico;

remover dados.

---

# Exclusão

Nunca apagar imediatamente.

Processo:

Solicitação

↓

Bloqueio

↓

Backup

↓

Anonimização/remoção

---

# Escalabilidade Futura

O modelo deve permitir evolução para:

Database por tenant

Schema por tenant

Infra dedicada

sem alterar domínio.

---

# Performance

Toda tabela principal deve possuir índice:

company_id

---

# Cache

Nunca criar cache compartilhado sem isolamento.

---

# Testes Obrigatórios

Sempre testar:

Empresa A não acessa Empresa B.

Usuário A não vê dados B.

IA A não utiliza conhecimento B.

Workflow A não executa contexto B.

---

# Regra Final

Multi-tenancy não é uma funcionalidade.

É uma propriedade fundamental do sistema.

Todo código deve preservar o isolamento entre empresas.
