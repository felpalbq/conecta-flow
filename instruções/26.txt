# Project Structure
Version: 1.0

---

# Objetivo

Definir a estrutura oficial do repositório principal.

O projeto utiliza organização Feature First.

---

# Princípio

Código deve ser organizado por domínio.

Nunca por tipo técnico.

---

# Estrutura raiz

```

conectaflow-app/

├── app/

├── features/

├── components/

├── lib/

├── services/

├── hooks/

├── types/

├── utils/

├── config/

├── integrations/

├── database/

├── docs/

├── tests/

└── public/

```

---

# app/

Responsável pela camada Next.js.

Contém:

- rotas
- layouts
- páginas
- providers

Não contém regra de negócio.

---

# features/

Centro da aplicação.

Cada domínio possui sua própria pasta.

Exemplo:

```

features/

 inbox/

 contacts/

 leads/

 dashboard/

 conversations/

 agent/

 appointments/

 payments/

 campaigns/

```

---

# Estrutura de uma Feature

Exemplo:

```

features/inbox/

components/

hooks/

services/

schemas/

types/

utils/

events/

```

---

# components/

Somente componentes compartilhados.

Exemplos:

Button

Modal

Table

Card

Nunca componentes específicos de domínio.

---

# lib/

Bibliotecas internas.

Exemplo:

supabase client

logger

permissions

analytics

---

# services/

Serviços globais.

Exemplo:

notification service

email service

storage service

---

# integrations/

Adaptadores externos.

Exemplo:

```

integrations/

 whatsapp/

 instagram/

 meta/

 anthropic/

 openai/

 stripe/

 google/

```

---

# database/

Tudo relacionado ao banco.

Inclui:

migrations

schemas

queries

types

---

# hooks/

Hooks realmente compartilhados.

Hooks específicos ficam dentro da feature.

---

# types/

Tipos globais.

Tipos de domínio permanecem dentro da feature.

---

# config/

Configurações:

environment

constants

feature flags

---

# docs/

Documentação oficial.

Nunca misturar documentação com código.

---

# tests/

Testes automatizados.

Organizados por domínio.

---

# Regras

Uma feature pode depender:

components

lib

services

types

---

Uma feature não deve depender diretamente:

de outra feature

---

Comunicação entre features:

através de serviços ou eventos.

---

# Exemplo

Inbox precisa de Lead.

Errado:

importar diretamente tabela de Lead.

Correto:

Lead Service

ou

Lead Event

---

# Regra para novos arquivos

Antes de criar perguntar:

Isso pertence a qual domínio?

Se não existe domínio.

Provavelmente está errado.

---

# Regra Final

A estrutura física do código deve refletir a estrutura mental do produto.
